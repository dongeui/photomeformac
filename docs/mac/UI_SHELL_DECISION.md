# Mac 앱 UI shell 결정

## 결론

Photome for Mac은 기본 UI를 기존 웹 대시보드/갤러리 WebView로 통합한다. 단, 앱 shell은 메뉴바 아이콘을 항상 제공해서 시작/중지/상태/대시보드 열기/폴더 선택 같은 Mac-native 제어를 맡긴다.

즉 선택지는 둘 중 하나가 아니라:

1. 메인 화면: 기존 웹 UI를 WebView로 표시
2. 우측 상단 메뉴바 아이콘: 런타임 제어/상태/빠른 액션

조합으로 간다.

## 이유

### WebView를 메인으로 쓰는 이유

1. 기존 웹 UI를 재사용하므로 Mac 전환 속도가 빠르다.
2. Docker판/서버판과 기능 차이가 생기지 않는다.
3. 대시보드, 갤러리, 검색, 사람, 상태 팝업을 중복 구현하지 않아도 된다.
4. 백엔드 API와 프론트 상태 로직을 그대로 검증할 수 있다.
5. 나중에 Docker/Windows 배포와 UI 일관성을 유지할 수 있다.

### 메뉴바 아이콘이 필요한 이유

1. Photome는 사진 서버/백그라운드 작업 성격이 강하다.
2. 사용자는 앱 창을 닫아도 스캔/AI 처리 상태를 보고 싶다.
3. Mac 사용자는 메뉴바에서 “실행 중/동기화 중/오류”를 보는 UX에 익숙하다.
4. 폴더 선택, LAN 공유 토글, 모델 다운로드, 대시보드 열기 같은 제어는 native 메뉴가 더 편하다.

## 초기 MVP UI 구조

### 메인 앱 창

- SwiftUI window
- 내부에 WebView
- URL: `http://127.0.0.1:<port>/dashboard`
- 백엔드 준비 전에는 로딩 화면 표시
- 백엔드 오류 시 재시작/로그 보기 버튼 표시

### 메뉴바 아이콘

상태 표시:

- 중지됨
- 시작 중
- 실행 중
- 스캔 중
- 이미지 AI 처리 중
- 오류

메뉴 항목:

1. Photome 열기
2. 대시보드 열기
3. 사진 폴더 선택
4. 스캔 시작
5. 이미지 AI 이어서 분석
6. LAN 공유 켜기/끄기
7. 로그 보기
8. 백엔드 재시작
9. 종료

MVP에서는 먼저 다음만 구현한다:

1. Photome 열기
2. 대시보드 열기
3. 백엔드 시작/중지 placeholder
4. 종료

## 하지 않을 것

초기에는 전체 웹 UI를 SwiftUI로 재작성하지 않는다. 그 방식은 개발량이 크고 Docker/서버판과 UI가 갈라진다.

초기에는 메뉴바만으로 모든 기능을 넣지 않는다. 메뉴바는 빠른 제어용이고, 실제 사진 탐색/검색/사람 관리는 WebView 메인 UI가 담당한다.

## 향후 개선

1. WebView 내 dashboard의 Mac app mode CSS/레이아웃 정리
2. 메뉴바 상태를 `/status` API와 연동
3. 폴더 선택은 NSOpenPanel로 받고 백엔드 settings/source-root API에 전달
4. LAN 공유 토글은 백엔드 bind/restart 계약 확정 후 구현
5. 모델 다운로드/AI 상태는 메뉴바와 WebView 양쪽에서 표시
6. 창 닫힘 시 백엔드는 계속 실행할지, 앱 종료와 함께 죽일지 설정 제공
