## 배경

얼굴 스택은 **YuNet(검출) + SFace 128d(임베딩)**, 매칭은 **코사인 ≥ 0.363**,
클러스터링은 **그리디 온라인 단일패스**(가까운 centroid에 임계값 넘으면 붙이고
아니면 새 인물, 자동 병합-백 없음 — `pipeline._resolve_person`). 경량 임베딩 +
이 전략 때문에 **같은 사람이 여러 클러스터로 과분할**되고, 배경/측면/소형 얼굴
(min 20px, 품질 게이트 없음)이 **싱글톤 노이즈 인물**을 양산한다. → 인물 600+.

본 플랜은 **모델/클러스터링 알고리즘은 그대로 두고**, 두 가지 UX 레버로 체감을
바꾼다: (1) 노이즈를 기본 숨김, (2) 과분할을 "병합 제안"으로 사용자 탭에 흡수.

관련 코드 앵커:
- `GET /people` 인물 목록 — `app/api/people.py:78` (현재 비병합·활성얼굴≥1 **전원**을
  face_count desc로 게이트·페이지네이션 없이 반환).
- 관리 페이지 `GET /people/manage` — `app/api/people.py:328`.
- 병합 `POST /people/merge` — `app/api/people.py:417` (이미 동작: 얼굴 재배정 +
  센트로이드 재계산).
- 센트로이드 파일 — `embeddings_root/people/v1/person-XXXXXX.json`
  (`person_centroids.person_centroid_path`).

## Item 1. 싱글톤·소형 클러스터 기본 숨김 + 사진수 정렬

**목표:** 기본 화면엔 의미 있는 인물만. 노이즈 싱글톤 600개는 접어둔다. 클러스터링
손 안 대고 "너무 많다" 체감 제거.

### 1.1 목록 API에 유의미성 게이트 — `status: TODO`

- `GET /people`에 파라미터 추가: `min_media`(기본 예: 2 또는 3), `include_minor=false`,
  선택적으로 `limit/offset`(페이지네이션).
- 응답을 두 묶음으로: **주요 인물**(media_count ≥ min_media) + **기타 카운트**
  (그 미만 인물 수). 정렬은 기존 face_count desc 유지.
- `having` 절에 media_count 조건만 추가하면 됨(이미 face_count로 그룹·정렬 중).
- **수용 기준:** 기본 호출이 주요 인물만 반환 + `minor_count` 제공. `include_minor=true`
  면 전원 반환(현행 호환).

### 1.2 관리 페이지 접힘 UI — `status: TODO`

- `/people/manage`에서 주요 인물 그리드 + 하단 **"기타 얼굴 N명 더 보기"** 토글
  (펼치면 1.1의 include_minor 호출).
- 이름 붙은/병합된 인물은 항상 주요 묶음으로(아래 Item-1 보너스).
- **수용 기준:** 첫 화면에 수백 개 싱글톤이 안 보이고, 한 번 펼쳐야 나온다.

### 1.3 (보너스) '확인됨' vs '추정' 구분 — `status: TODO`

- 사용자가 이름 붙였거나 병합한 인물 = "확인됨" 섹션 상단, 자동 무명 클러스터 =
  "추정/제안". 자동 결과를 *정답*이 아니라 *제안*으로 프레이밍.
- **수용 기준:** display_name이 `person-XXXXXX` 패턴이 아닌(=사용자 명명) 인물과
  그 외를 시각적으로 분리.

## Item 2. 병합 제안 큐 (임계값 바로 아래 밴드)

**목표:** 과분할이 병이면 병합이 약. 같은 사람일 확률 높은 클러스터 쌍을
"같은 사람인가요?"로 제시 → **원탭 합치기**로 사진 수십 장을 한 번에 정리.

### 2.1 병합 제안 산출 — `status: TODO`

- 새 엔드포인트 `GET /people/merge-suggestions`.
- 비병합 인물들의 센트로이드(이미 디스크에 있음)를 읽어 **쌍별 코사인** 계산,
  유사도가 **밴드 [low, threshold) (예: 0.30 ~ 0.363)** 에 드는 쌍을 desc로 반환.
  - 매칭 임계값(0.363)을 갓 못 넘은 쌍 = 같은 사람인데 갈라졌을 확률이 가장 높음.
- 각 제안에 양쪽 sample 얼굴 crop/카운트 포함(`/people/{id}/preview` 재사용).
- **성능:** O(n²) — n(비병합 인물) 캡(예: 상위 사진수 N명만 대상) + 결과 캐시 +
  유지보수 틱에서 사전계산 고려. 첫 버전은 동기 계산 + 캐시로 충분.
- **수용 기준:** 상위 후보 쌍이 유사도 높은 순으로 나오고 sample 얼굴이 붙는다.

### 2.2 제안 거절 저장 — `status: TODO`

- "다른 사람" 누른 쌍은 다시 제안하지 않게 **dismissed 쌍 저장**(정규화된
  (min_id,max_id) 키). 작은 테이블 또는 기존 피드백 저장소 재사용.
- **수용 기준:** 거절한 쌍이 새로고침/다음 산출에서 재등장하지 않음.

### 2.3 리뷰 UI — `status: TODO`

- `/people/manage`(또는 전용 탭)에 "같은 사람인가요?" 큐: 쌍을 나란히, **"같은 사람"**
  (→ `POST /people/merge`) / **"다름"**(→ 2.2 dismiss) 원탭. 처리 후 다음 쌍.
- **수용 기준:** 탭 한 번에 병합 완료 + 큐가 다음으로 넘어감. 병합은 기존
  centroid 재계산까지 자동(이미 구현).

## 참고

근본은 경량 임베딩(SFace) + 그리디 온라인 클러스터링의 과분할이다. 모델 천장과
클러스터링 전략은 별개 레버이며, 체감 비중은 후자가 크다. 본 플랜은 그 둘을
건드리지 않고 **클러스터 수/품질을 UX에서 흡수**하는 것이 목적이다.
